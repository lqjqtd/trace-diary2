# 素履（Trace）天气功能开发日志

**日期**：2026-06-29  
**功能模块**：日记天气（自动获取 + 手动切换 + 温度显示）

---

## 一、背景与需求

### 功能目标
在位置功能的基础上，增加天气自动获取和显示功能，让日记记录更加完整。

### 用户需求
1. 定位/选位置后自动获取当天天气
2. 扩展天气类型（雾、霾、阴等）
3. 点击天气图标可手动切换
4. 天气获取失败时静默处理，不打扰用户
5. 去掉编辑器中原先单独的天气选择栏，改为位置旁显示
6. 首页列表和详情页显示温度和位置信息

### 补写日记场景
- 存在补写日记的情况（如5月6号写5月1号的日记）
- 方案A（本期实现）：仅当天日记自动获取实时天气，历史日期不自动获取
- 方案B（待办）：支持历史天气查询，需对接第三方历史天气API

---

## 二、技术方案

### 方案A（本期实现）
- 使用高德地图「天气查询」API 获取实时天气
- 仅当天日记自动获取天气，历史日期不自动获取
- 用户可手动切换天气，手动切换后不再被自动获取覆盖
- 天气获取失败时静默处理，不显示错误提示

### 方案B（待办，后续实现）
- 对接支持历史天气查询的第三方API
- 支持按日期+位置查询任意日期的天气
- 增加天气缓存机制，减少重复请求

## 五、关键技术点

### 5.1 天气类型映射策略

采用「精确匹配 + 模糊匹配」两级策略：
1. 先查 WEATHER_MAP 精确匹配表（30+ 种高德天气现象）
2. 匹配失败则按关键词模糊匹配（晴/雷/雪/雨/阴/云/雾/霾/沙/风）

保证了高德返回的各种天气现象都能正确归类。

### 5.2 自动获取的防覆盖机制

使用 `weatherManuallySetRef` 引用标记：
- 初始值为 false，允许自动获取
- 用户手动选择/清除天气时设为 true
- 自动获取 useEffect 中检查 ref，为 true 则跳过
- 确保用户手动操作优先级高于自动获取

### 5.3 静默失败设计

天气是锦上添花的功能，获取失败不应该打扰用户：
- try/catch 包裹所有天气 API 调用
- catch 块不做任何提示，仅 console.error 日志
- UI 上保持天气占位符，用户可手动点击填写

### 5.4 仅当天自动获取

为什么方案A只做当天：
- 高德天气 API 只有实时天气接口，没有历史天气
- 补写历史日记时，获取当天的天气是错误的
- 历史天气需要对接其他 API（方案B），留待后续实现

## 六、高德开放平台配置

### 使用的 API
- **天气查询 API**：`https://restapi.amap.com/v3/weather/weatherInfo`
- **逆地理编码 API**：`https://restapi.amap.com/v3/geocode/regeo`（经纬度转城市编码）

### 复用已有的 Web 服务 Key
- 与位置搜索、逆地理编码共用同一个 Web Key
- 不需要额外申请新的 Key
